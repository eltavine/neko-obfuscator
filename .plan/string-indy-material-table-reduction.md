# String/Indy Material Table Reduction Plan

## Objective

Reduce repeated JVM transform helper/material structures by moving repeated
string and invoke-dynamic material into class-level `<clinit>`-registered
object-array tables, without weakening obfuscation, changing JVM ABI behavior,
or introducing fallback paths.

This document is the active implementation todo and evidence log for this
work. Source implementation has started; unchecked or in-progress rows remain
outside completion.

## Global constraints

- Do not implement a sample-, benchmark-, class-, method-, descriptor-, or
  crash-site-specific fix.
- Preserve all hidden key parameters, packed `Object[]` parameter carriers,
  dynamic key propagation, CFF semantics, and transform compatibility.
- Do not add JNI fallback, original-bytecode fallback, skip-on-error behavior,
  Java bridge methods, runtime adapter methods, or helper classes.
- Do not expose plaintext names, descriptors, resource paths, key material, or
  obfuscation seeds through the new material tables.
- Repository `./gradlew` must be used only after explicit permission.
- A subtask is not complete until freshly generated artifacts exercise the
  changed path and the matching todo item is updated in the same commit.

## Runtime target rows

- `R-build`: regenerate affected artifacts with repository `./gradlew`.
- `R-test`: run the generated TEST native jar.
- `R-obfusjack`: run the obfusjack Java fixture.
- `R-native-obfusjack`: run the generated obfusjack native jar.
- `R-native-test`: run the generated TEST native jar under the native path.
- `R-inspect`: inspect generated bytecode/C/native output for forbidden JNI,
  fallback markers, plaintext protected material, and duplicate helper growth.
- `R-negative`: prove missing required symbols/capabilities fail closed instead
  of falling back, if the implementation touches resolver or native capability
  binding.

## Planned bytecode/data shape

This section fixes the intended generated shape for later implementation. If
implementation evidence proves one detail verifier-invalid, update this plan
before changing source.

String payload material table:

- Field shape: one synthetic/private/static per-class field with descriptor
  `[Ljava/lang/Object;`.
- Cell shape: each populated cell stores one encrypted payload byte array with
  runtime type `[B`.
- Slot meaning: a slot index identifies payload location only. It does not
  identify, encode, or replace the final decode key.
- Initialization: the existing class initialization injection surface allocates
  the table once and stores encrypted byte arrays into their assigned slots.
- Runtime access: transformed string use loads the table, loads the site slot,
  performs `AALOAD`, casts to `[B`, and passes the bytes into the decode path.
- Empty classes: classes with no transformed string payload sites must not
  receive this table.

Key material object table:

- Field shape: one synthetic/private/static per-class field with descriptor
  `[Ljava/lang/Object;`, separate from the string payload table.
- Cell shape: each populated cell stores encoded/intermediate key material, for
  example protected primitive-array words or an equivalent transform-internal
  object shape already valid for generated bytecode.
- Slot meaning: a slot selects an encoded key-material family. It never selects
  a directly usable final key.
- Initialization: `<clinit>` computes and stores encoded cells using protected
  class-local material. Raw static keys must not be stored as direct constants
  or directly usable object values.
- Runtime access: helpers or dispatch paths load the table cell and mix it with
  live method-entry/CFF/key-dispatch state before use.
- Mutation: CFF/key-dispatch paths may re-encode or update cells only with live
  state input and nonlinear operations; no inverse-pair, neutral, or dead-store
  updates are allowed.
- Strong dynamic dependency rule: table cells, slot indexes, site seeds,
  class-init material, or helper constants must never be sufficient to decrypt
  string payloads. Decode roots, cipher keys, byte-layer reversal, and plaintext
  unmasking must depend on live flowkey/CFF state.

String decode helper shape:

- Phase 1 decode may keep the existing helper structure while replacing payload
  field loads with string material table loads.
- After the key table is introduced, every string decode helper variant in the
  changed path must either receive the key table as an explicit argument or load
  the per-class key table field directly. The implementation must pick one
  shape for the changed class and record bytecode/verifier evidence.
- Decode helpers must consume key-table cells as mixed dynamic inputs. They
  must not embed per-helper static key constants that recreate the current
  duplicated-helper problem.
- Existing cache/fingerprint behavior may remain unchanged until separate
  evidence proves it can be safely table-backed or consolidated.

Verifier and ABI constraints:

- All generated stack shapes must remain verifier-valid for normal methods,
  constructors, exception paths, synchronized methods, loops, recursion,
  lambdas, MethodHandle paths, reflection paths, virtual/interface dispatch,
  and abstract dispatch.
- Existing external ABI entry points remain valid. Constructors, class
  initializers, `main([Ljava/lang/String;)V`, and inherited external API
  override slots must not receive incompatible descriptor changes.
- Hidden key parameters and packed `Object[]` carriers remain present and
  semantically live; the tables provide shared material, not ABI downgrades.

- This document is updated with the concrete evidence chain.
- No source files are changed.
- Commit contains only this plan update if a commit is requested for this

Evidence chain recorded:

- `JvmStringObfuscationPass.ensureStringSiteCache` is the current string-site
  allocation point. It creates a distinct site cache for each transformed
  string use instead of registering shared encrypted payload material in a
  class-level table.
- `JvmStringObfuscationPass.installPayloadInit` is the current per-site payload
  material path. It installs a separate payload init helper and payload field
  for each site cache, so many structurally identical encrypted-byte payload
  holders are emitted into heavily obfuscated classes.
- `JvmStringObfuscationPass.installDecodeHelper` is the current per-site decode
  helper path. Its helper shape is repeated with site-specific static material,
  which increases method/field count and class size without adding live dynamic
  dispatch complexity.
- `JvmStringObfuscationPass.emitDecodedString` is the current runtime use path.
  The transformed call site obtains the encrypted payload through a site payload
  field and then invokes the generated decode path. This proves the first safe
  movement boundary: encrypted payload bytes can be registered in a class-level
  object table while preserving decode semantics, as long as the runtime lookup
  still supplies the same protected payload material and remains keyed by live
  obfuscated state.
- `StringSiteCache` currently carries per-site payload names/material. This is
  the data structure boundary that must become index/table oriented for string
  payload material, rather than field/helper oriented per call site.
- `JvmInvokeDynamicObfuscationPass.ensureHelpers` already creates the main indy
  helper set once per class. Bootstrap, resolver, decode, mix, flow, filter,
  and cache structures are already class-local shared helpers rather than purely
  per-site helper methods.
- Indy therefore needs a separate assessment before source changes: only
  remaining duplicated indy material should move into a class-level table. The
  existing class-level helper sharing must not be replaced with Java bridges,
  fallback adapters, or descriptor-only/static-key recomputation.

Invariant:

- The duplicated structure proven by static inspection is string encrypted
  payload/helper material emitted per site, while the runtime semantic input is
  the protected payload plus live key-dependent decode state. Moving only the
  encrypted payload material behind a class-level indexed object table preserves
  the runtime data dependency and creates a stronger future surface for dynamic
  table mutation, provided the table index does not become a plaintext key leak
  or a replacement for live key flow.

- `R-test`
- `R-obfusjack`
- `R-native-test`
- `R-inspect`

Deferred implementation design:

- Add a per-class key table field with descriptor `[Ljava/lang/Object;` for
  classes that require shared helper/key material. The table is separate from
  the string payload material table so payload storage and key-state mutation
  can be validated independently.
- Store each key cell as an opaque object cell, such as a protected primitive
  array or equivalent internal cell object already allowed by the transform

  plaintext keys.
- Initialize the key table from the existing class initialization injection
  surface. `<clinit>` computes encoded cells using class-local protected
  material and stores them into stable slots.
- Do not let a table slot index identify a final key. A slot index only selects

  method-entry key material, CFF state, or downstream dynamic dispatch state.

  class-field load, choosing the smaller verifier-safe shape during
  implementation evidence collection. The helper must use table cells as
  dynamic mix inputs, not as static replacement constants.
- Update CFF/key-dispatch integration so selected key cells are mutated or
  re-encoded from live state during dispatcher transitions. Mutation formulas
  must use nonlinear operations and live external input, and must not contain
  inverse-pair/self-canceling updates.
- Keep hidden method keys and packed `Object[]` parameter carriers intact. The
  key table augments key transfer and helper material lookup; it does not
  replace required ABI-obfuscation surfaces.
- If a required key-table cell, live key input, or update path cannot be proven
  for a helper family, leave that helper family unchanged and record the
  evidence gap. Do not introduce fallback behavior or static-key recomputation.

- Do not change CFF block boundaries, block count, or block selection.
- Do not move unrelated payload bytes into the key table.
- Do not add Java bridge methods, adapter methods, helper classes, or runtime

- Do not make LambdaMetafactory, MethodHandle, reflection, virtual/interface

- Generated bytecode has one shared key table per affected class rather than

- String helper bytecode reads key material from the key object array and mixes

- CFF/key-dispatch bytecode contains key-table update operations that depend on

Implementation state:

- Source implementation has been applied for string decode paths in
  `JvmStringObfuscationPass`.
- A per-class synthetic static key material `Object[]` is emitted alongside the
  string payload material table.
- The key table is initialized from a `__neko_strkey$...` class-init helper for
  normal classes and through direct `<clinit>` assignment for interfaces. Each
  cell is an encoded `int[]` material cell derived from class/site seed
  material, not a directly usable final decode key.
- Compile-time encryption and runtime decode both use the same key-cell mixed
  root formula. The runtime helper loads the key cell with `AALOAD`, casts it
  to `[I`, and mixes it with the live CFF-derived root before fingerprint,
  cipher key generation, byte-layer reversal, and XOR plaintext unmasking.
- Key material cells are stored as epoch-masked encoded `int[]` cells. Decode
  helpers recover canonical key material through the current cell epoch, then
  derive a new epoch from the live flowkey/CFF-derived root and re-encode all
  key words back into the cell. The table cell mutates while preserving decode
  semantics, and the cell alone is not sufficient to decrypt payload bytes.
- The key material table field is registered with the existing generated-field
  reflection filter.
