# CFF Hot Key-Update Low-Stack Execution Plan - 2026-05-27

## Scope

Optimize one generic JVM CFF hot-path key-update surface: the generated
method-key-from-state helper and the matching no-active-table inline method-key
shape. The current accepted helper is blocked by size; prior size-only
fallback removal exposed a stack-depth blocker. The implementation must address
both surfaces without changing the formula or ABI.

This is a standalone plan. It does not amend or continue any existing `.plan/`
document.

The change must preserve:

- JVM ABI compatibility.
- Full CFF block coverage, block boundaries, fake rows, poison rows, hard-fail
  rows, material rows, and dispatch token rows.
- Hidden method-key propagation and packed `Object[]` carrier ABI.
- Reflection, MethodHandle, LambdaMetafactory, invokedynamic, constructor, and
  dynamic invocation compatibility.
- The method-key helper descriptor `(IIIIJJ)J`.
- The masked salt-pair callsite ABI: `methodSalt ^ saltMask` and `saltMask`.
- The raw method-key formula and live `guard/path/block/pc` inputs.

Forbidden in this plan:

- Benchmark, fixture, class, method, owner, descriptor, or hot-log
  special-casing.
- Static-key recomputation, raw method-salt exposure, fallback to original
  bytecode, skip-on-error behavior, helper-layer compatibility bypasses, or new
  Java runtime helper layers.
- Changes to token-material rows, transition-material rows, step-material rows,
  key-transfer rows, island-material rows, CFF block construction, CFF block
  selection, CFF block granularity, or outliner thresholds.
- Writing to existing JVM performance plans.

This plan is not a retry of rejected routes:

- Not P4.4 fallback removal alone: fresh HKU-1 accepted-source evidence starts
  from the current `callee is too large` blocker, and historical P4.4 evidence

  transition row pre-scaling, transition `out[]` carrier replacement, row-pair
  caching, material-carrier caching, or step-material mask caching.

## Invariants

Raw method-key formula:

```text
(((long) guardKey) << 32 ^ (((long) pathKey) & 0xFFFFFFFFL))
+ (((long) blockKey) ^ methodSalt)
^ ((((long) pcToken) & 0xFFFFFFFFL) * METHOD_KEY_PC_MIX)
```

Runtime helper inputs:

```text
guard, path, block, pc, methodSalt ^ saltMask, saltMask
```

The salt is recovered only as:

```text
(methodSalt ^ saltMask) ^ saltMask
```

Generation-time raw-nonzero proof:

- If a generated state's raw formula is already nonzero, no state changes.
- If a generated state's raw formula is zero, adjust only `methodSalt`.
- Toggling bit `0` of `methodSalt` changes `((long) blockKey) ^ methodSalt` by
  one modulo `2^64`.
- If toggling bit `0` would make the salt zero, toggling bit `1` keeps the salt
  nonzero and changes that term by two modulo `2^64`.
- Because the previous raw value was zero only when the summed state term
  equaled the pc-mix term, the adjusted salt yields a nonzero raw method key.
- The masked salt-pair callsite remains masked; raw method salt is not exposed.

Low-stack expression proof:

- Reordering through helper-local long temporaries changes only JVM operand
  stack shape.
- The same live `guard/path/block/pc` inputs and the same recovered salt feed
  the same raw formula.
- No CFF state, row, material, dispatcher, or callsite identity changes.
